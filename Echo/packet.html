Field-by-field analysis
SEQ — Sequence Number shrink 4B→2B
Detects lost/out-of-order packets. 4 bytes supports 4 billion packets = 40,000 seconds at 100 pkt/s. You only need 2 bytes (65535 wrap-around every ~109 minutes). Safe to halve.
TS — Timestamp µs shrink 8B→4B
Used for delay measurement. 8 bytes stores absolute epoch time. A 4B relative timestamp (ms since session start) is enough — gives 49 days of range. Saves 4 bytes per packet.
NS — Num Samples drop
Tells receiver how many samples are in payload. But since frame size is fixed at 160 samples (agreed at startup), receiver can infer it from UDP packet length. Redundant — safe to remove.
SR — Sample Rate drop
Carries 16000 in every single packet — same value every time. This never changes during a session. Move it to a one-time handshake UDP message at connect. Completely eliminates 2B per packet.
PCM Payload — 320 bytes keep as-is
160 samples × 2 bytes = 320 bytes. This IS the audio. You cannot reduce this without compression (e.g. µ-law, ADPCM, Opus). For this prototype, raw PCM is correct — compression adds CPU overhead and complexity with diminishing returns on localhost. If you needed to go over a low-bandwidth WAN, replacing int16 PCM with 8-bit µ-law would halve the payload to 160 bytes.
